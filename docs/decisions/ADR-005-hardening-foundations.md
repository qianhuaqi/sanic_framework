# ADR-005: Identifiers, errors, configuration, serialization, Runtime Record, and telemetry

- Status: Proposed
- Date: 2026-06-28
- Parent architecture Issue: #25
- Decision Issue: #43

## Context

LingShu has accepted repository governance, runtime concurrency, package boundaries, and the Application Kernel/request pipeline. P1 still cannot begin until cross-cutting hardening semantics are fixed. Identifiers, errors, configuration, serialization, records, and telemetry must use one coherent model or every component will invent incompatible formats and failure behavior.

## Decision

### 1. Unified time model

LingShu uses two distinct clocks:

- UTC wall-clock time for human-readable timestamps and cross-process correlation;
- monotonic time for Deadline, timeout, scheduling, queue wait, duration, and local event ordering.

Wall timestamps use RFC3339 UTC with a trailing `Z`. Durations use integer nanoseconds where practical. A monotonic value is meaningful only inside the process and boot instance that produced it; it is never compared across Workers or machines.

Runtime events include a per-record monotonically increasing sequence number. Cross-process events have no fabricated total order.

### 2. Identifier types

LingShu defines distinct typed identifiers:

- `RequestId`;
- `ConnectionId`;
- `TraceId`;
- `OperationId`;
- `WorkerId`;
- `RecordId`;
- `RevisionId`.

Runtime identifiers are opaque, non-semantic, unpredictable 128-bit values generated by a cryptographically secure source. Their canonical text form is 32 lowercase hexadecimal characters without separators.

`RevisionId` is different: it is a lowercase hexadecimal SHA-256 digest of the canonical validated Application Revision and is 64 characters.

Rules:

- identifier values contain no timestamp, host, PID, tenant, route, user, or business meaning;
- typed identifiers are not interchangeable merely because their text shape matches;
- generation failure is fatal for the affected admission/startup operation;
- IDs are safe for logs and record keys but are not authorization credentials;
- IDs are never used directly as unvalidated user-controlled filesystem paths;
- internal IDs remain stable for their Scope and are immutable.

### 3. Identifier propagation and trust

LingShu always creates its own internal `RequestId`, `ConnectionId`, `OperationId`, `WorkerId`, and `RecordId`.

An inbound `X-Request-ID` is treated only as `external_request_id` after length and character validation. It never replaces the internal Request ID. The response emits the internal Request ID through `X-Request-ID` unless a later protocol decision changes the header name.

A valid distributed trace context may continue a remote Trace ID, but it is propagation metadata, not trusted identity or authorization input. Invalid trace context is ignored or rejected only under explicit strict policy and is recorded safely.

Every outbound framework integration receives the current trace context and may receive the internal Request ID as correlation metadata. Secret or tenant context is not propagated implicitly.

### 4. Exception taxonomy

`LingShuError` is the root for ordinary framework failures. The stable categories are:

- `ConfigurationError`;
- `LifecycleError`;
- `ProtocolError`;
- `RequestError`;
- `RoutingError`;
- `HandlerContractError`;
- `SerializationError`;
- `ResourceLimitError`;
- `AdmissionError`;
- `DeadlineError`;
- `ExtensionError`;
- `RecordError`;
- `StorageError`;
- `InternalError`.

Cancellation remains a control-flow signal governed by ADR-002 and is not converted into an ordinary `LingShuError` by broad exception handling.

Each framework error conceptually carries:

```text
code
safe_message
client_visible
retryable
http_status?
severity
fatal_scope
safe_details?
internal_cause?
```

### 5. Stable error codes and client safety

Error codes are stable lowercase dotted identifiers, for example:

```text
config.invalid
lifecycle.invalid_state
request.body_too_large
route.not_found
handler.invalid_return
serialization.invalid_json
resource.capacity_exhausted
record.storage_unavailable
internal.error
```

A code changes only when semantics change. Human wording may be improved without changing the code.

Client-visible framework errors use `application/problem+json` and contain only allowlisted safe fields:

```json
{
  "type": "urn:lingshu:error:request.body_too_large",
  "title": "Request body too large",
  "status": 413,
  "detail": "The request body exceeds the configured limit.",
  "instance": "urn:lingshu:request:<request_id>",
  "code": "request.body_too_large",
  "request_id": "<internal request id>"
}
```

Optional `details` must be schema-defined and redacted. Stack traces, exception repr, absolute paths, source code, configuration values, environment values, secrets, SQL, credentials, request bodies, and internal topology are never exposed by default.

Unexpected exceptions map to `internal.error` with a generic client message. Full causes remain internal and are subject to record redaction.

### 6. Cause, retry, severity, and fatal scope

Errors preserve an internal cause chain without serializing it blindly.

`retryable` describes technical retry suitability, not whether the framework automatically retries. Automatic retry is a separate policy decision.

`fatal_scope` is one of:

```text
operation
request
connection
worker
supervisor
```

Severity is one of `debug`, `info`, `warning`, `error`, or `critical`. Fatal scope and severity are determined by the error type and context, not by client input.

### 7. Configuration sources and precedence

Configuration precedence is deterministic:

```text
built-in defaults
< configuration file
< environment variables
< CLI overrides
< explicit programmatic overrides
```

Rules:

- every value is normalized through a declared schema;
- unknown keys fail by default;
- duplicate normalized keys in one source fail;
- nested mappings merge by key; scalar and sequence values replace rather than concatenate;
- environment parsing is schema-driven and never uses `eval`;
- file configuration declares a schema version;
- a schema-version mismatch fails fast unless an explicit migration exists;
- migration is explicit, version-to-version, testable, and never silently lossy.

The exact configuration file syntax is deferred.

### 8. Secrets and configuration snapshots

Secret values use a dedicated secret type or provider handle.

- secret values redact their `repr`, log, telemetry, error, and diagnostic output;
- secrets are not included in Revision hashing as plaintext; a stable redacted/provider identity is used where required;
- secret equality and change detection avoid exposing values;
- dumping a configuration snapshot excludes or masks secrets;
- missing secrets fail before readiness.

The running application receives an immutable typed Configuration Snapshot. Mutable source dictionaries are not retained as runtime configuration.

### 9. Configuration reload and rollback

Reload is revision-based:

```text
load
→ normalize
→ validate schema
→ resolve secrets
→ prepare resources
→ compile/freeze new Application Revision
→ publish atomically
→ drain old Revision
→ cleanup old resources
```

If any pre-publication step fails, the old revision remains active and unchanged. Post-publication failure triggers rollback when safe; otherwise the service enters a declared degraded/not-ready state and records the divergence.

No component observes a partially applied configuration. Multi-Worker rollout must preserve revision identity and report partial rollout explicitly; its transport/coordinator implementation remains deferred.

### 10. Serialization registry

Serialization is explicit and media-type based. Core does not use arbitrary object introspection as a wire contract.

The baseline serializers are:

- text: `text/plain; charset=utf-8`;
- binary: `application/octet-stream`;
- JSON: `application/json` encoded as UTF-8;
- framework error: `application/problem+json` encoded as UTF-8.

A serializer declares supported Python value types, media types, encode/decode functions, safety limits, and determinism characteristics. Unknown values fail with `serialization.unsupported_type`.

### 11. JSON rules

LingShu JSON behavior is strict:

- UTF-8 only; a conflicting unsupported charset is rejected;
- no BOM on output;
- duplicate object keys are rejected on input;
- NaN, positive infinity, and negative infinity are rejected on input and output;
- maximum body size, nesting depth, container items, string length, and numeric token length are bounded;
- unknown Python objects are not serialized automatically;
- `None` maps to JSON `null`;
- booleans, strings, finite numbers, lists, and string-keyed mappings are supported;
- datetime uses RFC3339 UTC only through an explicit serializer;
- bytes use base64 only through an explicit schema/serializer and never by surprise;
- Decimal and UUID-like values require explicit serializers;
- normal response JSON preserves input mapping order; canonical sorted-key JSON is used only where hashing or deterministic records require it.

### 12. Content negotiation

For request decoding:

- an unsupported media type produces 415;
- a structured decode request with a missing required Content-Type produces 415;
- media-type parameters are parsed strictly;
- route body policy decides whether a body is allowed, required, streamed, or decoded.

For response selection:

- absent `Accept` behaves as `*/*`;
- quality values and specificity determine the chosen registered representation;
- no acceptable representation produces 406;
- `Content-Type` is emitted explicitly;
- content sniffing is not used.

Full form, multipart, upload, compression, and streaming serialization formats are deferred.

### 13. Runtime Record logical model

Every admitted business request reserves a `RecordId` and opens one logical append-only Runtime Record before Handler execution.

If the required record capacity cannot be reserved, the request is rejected before business handling. A request is never reported as fully audited when no record was opened.

The record contains versioned events rather than one mutable document.

### 14. Runtime Record event envelope

Every event contains at least:

```text
schema_version
event_type
event_sequence
wall_time
monotonic_ns
component
outcome
record_id
request_id
connection_id?
trace_id?
operation_id?
worker_id
revision_id
route_name?
http_method?
http_status?
error_code?
cancellation_reason?
duration_ns?
attributes
truncated
```

`event_sequence` is strictly increasing within one Record. Attributes are allowlisted, bounded, and redacted. Large bodies and arbitrary objects are never embedded by default; approved payloads use bounded summaries or external references.

### 15. Default local storage format

The default local writer uses versioned UTF-8 JSON Lines in append-only rotated segment files.

- one event occupies one line;
- incomplete trailing lines after a crash are discarded during recovery;
- active and closed segment state is tracked by an atomic manifest/index;
- manifest changes use temporary-file write, flush, and atomic rename where supported;
- segment names are framework-generated and contain only safe internal identifiers;
- writer paths remain under a configured canonical base directory;
- symlink/path traversal and unexpected ownership/permission conditions are rejected;
- external database/object-storage exporters are optional extensions.

The exact segment size and rotation interval are configuration values decided during implementation tuning.

### 16. Durability and failure modes

The writer supports declared durability policies:

```text
buffered
flush
fsync
```

The selected policy is visible in diagnostics and records. No policy claims stronger durability than it provides.

Record queues are bounded. Writer failure produces explicit counters, health state, and emergency diagnostics.

Two service policies are supported:

- `required`: reject new business requests when a record cannot be reserved or safely queued;
- `best_effort`: continue service while explicitly marking drops/incomplete records.

Because request-level auditability is a confirmed LingShu invariant, the default business-request policy is `required`. Health and shutdown endpoints may use separately declared minimal-record behavior.

### 17. Budgets, watermarks, and retention

Runtime Record configuration has independent limits for:

- event bytes;
- record bytes and event count;
- queue items and queue bytes;
- active segment bytes;
- total local storage bytes;
- retention age;
- cleanup work per cycle;
- flush and shutdown time.

Disk behavior uses watermarks:

- below soft watermark: normal capture;
- above soft watermark: reduce optional detail, truncate approved payload summaries, accelerate cleanup, and emit warning health;
- at hard watermark: reject new `required` requests and mark not ready;
- at critical reserve: preserve only minimal failure/health/shutdown diagnostics and protect filesystem stability.

Closed segments are deleted only by the retention process after age/reference checks. Active segments are never removed by cleanup.

### 18. Crash recovery

On startup the writer:

1. acquires a writer-instance lock/lease;
2. validates the base path and permissions;
3. loads or rebuilds the segment manifest;
4. scans active tails;
5. truncates incomplete final lines;
6. validates envelope versions and required fields;
7. marks unrecoverable segments quarantined rather than silently deleting them;
8. reports recovered, truncated, quarantined, and lost-event estimates;
9. becomes ready only when the selected record policy can be honored.

Recovery work is bounded. Excessive recovery time leaves the service not ready rather than blocking indefinitely.

### 19. Telemetry common fields

Structured logs, traces, Runtime Records, and diagnostic events share common names where applicable:

```text
timestamp
component
event
severity
outcome
framework_version
revision_id
worker_id
connection_id
request_id
trace_id
operation_id
route_name
http_method
http_status
error_code
retryable
cancellation_reason
duration_ns
```

Fields that do not apply are omitted rather than emitted as misleading empty values.

Request, trace, connection, operation, and record identifiers are high-cardinality and must not be used as metric label dimensions. Metrics use bounded dimensions such as component, route template/name, method, status class, outcome, and error code.

### 20. Redaction and cardinality

All observability paths use one redaction classification:

```text
public
internal
sensitive
secret
```

- `secret` is never emitted;
- `sensitive` is omitted, hashed, tokenized, or truncated only by explicit policy;
- headers, query values, body values, cookies, credentials, tokens, and configuration secrets are sensitive by default;
- route templates may be telemetry dimensions; raw paths are not metric labels;
- arbitrary exception messages are internal and not stable telemetry dimensions;
- attributes and labels have count, name-length, and value-length limits.

### 21. Required tests

Implementation acceptance includes:

- identifier uniqueness, canonical form, typed non-interchangeability, and secure-source failure;
- inbound correlation validation and internal Request ID replacement protection;
- stable error-code catalog and safe problem responses;
- cause-chain redaction and cancellation preservation;
- deterministic configuration precedence and unknown-key failure;
- schema migration, secret redaction, immutable snapshots, reload success, failed prepare, and rollback;
- strict JSON duplicate-key, non-finite-number, depth, size, and unsupported-type behavior;
- 406/415 negotiation paths;
- Runtime Record reservation before Handler execution;
- bounded queue saturation under `required` and `best_effort` policies;
- partial-line, damaged-manifest, permission, disk-full, watermark, retention, and crash-recovery tests;
- common telemetry-field consistency and absence of high-cardinality metric labels;
- no secret leakage through logs, errors, records, metrics, traces, diagnostics, or configuration dumps.

## Rejected alternatives

- trusting inbound Request IDs as internal IDs;
- semantic or sequential externally guessable IDs;
- exposing raw exception messages or tracebacks to clients;
- unversioned error codes and error envelopes;
- mutable runtime configuration dictionaries;
- partial in-place reload;
- silent schema migration;
- arbitrary-object JSON serialization;
- permissive NaN/Infinity or duplicate JSON keys;
- unbounded record queues or disk usage;
- deleting active record segments during cleanup;
- claiming complete audit after record reservation/write failure;
- using request/trace IDs as metric labels;
- treating wall-clock time as a Deadline clock.

## Intentionally deferred

- exact default numeric limits and retention duration;
- configuration file syntax and secret-provider implementations;
- multi-Worker reload transport and consensus mechanism;
- full form, multipart, upload, compression, and streaming formats;
- concrete logging, metrics, tracing, database, or object-storage backends;
- official OpenTelemetry integration;
- cross-machine clock synchronization policy;
- post-v1.0 error-code compatibility policy.
